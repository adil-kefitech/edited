3. The Real Problem — The Bug in AddNewBatchView
The current duplicate validation is:

python
if year_master:
    batch_exists = Batch.objects.filter(academic_year=year_master).exists()
    if batch_exists:
        return error(f"A batch for the Admission Year {year_int} already exists!")
This check is too broad. It blocks ANY second batch for the same admission year purely based on year, ignoring the title. In a university context you will have multiple batches under the same year — for example BCA 2024, BSc CS 2024, or regular vs lateral entry. Once the first batch for year 2024 is created, this wall shuts the door on every subsequent batch for 2024.

The correct check should guard against the same title under the same year — not just the year:

python
if year_master:
    title_exists = Batch.objects.filter(
        academic_year=year_master,
        title__iexact=title          # ← add the title condition
    ).exists()
    if title_exists:
        return error(f"A batch named '{title}' for Admission Year {year_int} already exists!")
4. Secondary Issue — UX After Batch Creation
After AddNewBatchView returns success, the JS in add_batch.html simply resets the form. The user has no indication to go to student mapping and no way to do it in one step. The batch list in student_mapping.html is server-rendered at page-load, so the new batch won't appear unless the user manually navigates there and reloads.

The fix: on success, offer a "Go to Student Mapping" button in the SweetAlert so the user is taken directly where they need to go next.

One Thing You Should Verify
The only real prerequisite for this whole chain to work is that the StudentDetail records for the new 1st semester students have their admission_year FK pointing to the same AdmissionYearMaster that the new batch points to.

If students were imported/uploaded before the batch was created, verify:

python
# The batch created by AddNewBatchView
Batch.academic_year  →  AdmissionYearMaster(admission_year=2025)

# The students' record
StudentDetail.admission_year  →  AdmissionYearMaster(admission_year=2025)
Both must point to the same AdmissionYearMaster row (same id, not just the same year integer). If student import creates a separate AdmissionYearMaster entry for the same year, the FK match admission_year=batch.academic_year will return zero students even though the year number looks identical. That's the one silent failure point in this entire system.
